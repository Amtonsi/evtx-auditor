# Проектирование EVTX Аудитора Windows

Дата: 4 июля 2026 года

## 1. Назначение

EVTX Аудитор — автономное настольное приложение для Windows, которое анализирует журналы событий из ZIP-архивов и формирует единый подробный HTML-отчёт по всем обнаруженным узлам.

Исходный набор данных находится в `C:\Users\impal\Downloads\АУДИТ 2`. На момент проектирования в нём находятся 22 каталога узлов, 22 ZIP-архива и 66 журналов: по одному `Application.evtx`, `Security.evtx` и `System.evtx` для каждого узла.

Приложение должно:

- работать через графическое окно PySide6;
- не изменять исходные архивы;
- анализировать последние 30 дней относительно самого позднего события каждого узла;
- находить события Windows уровней Critical и Error;
- дополнительно выявлять опасные события информационной безопасности;
- объединять повторяющиеся события, сохраняя доказательства;
- продолжать работу при повреждении отдельного архива или журнала;
- создавать один автономный HTML-файл с общей статистикой и разделом каждого узла;
- автоматически открывать отчёт после успешного создания;
- поставляться в виде готового `EVTXAuditor.exe` и полного исходного кода.

## 2. Границы проекта

### Входит в первую версию

- ZIP-архивы, содержащие файлы `.evtx`.
- Журналы Application, Security и System.
- Автоматическое определение узла по имени родительского каталога, имени архива и полю `Computer` внутри события.
- Локальный анализ без отправки журналов в сеть.
- Поиск, фильтрация и навигация внутри автономного HTML.
- Русские названия, пояснения и рекомендации для известного каталога событий.
- Технические поля для всех найденных событий, даже если человекочитаемое описание поставщика недоступно.
- Сборка Windows x64 через PyInstaller.

### Не входит в первую версию

- Подключение к удалённым компьютерам и чтение их активных журналов.
- Изменение настроек Windows или автоматическое исправление найденных проблем.
- Отправка отчётов по почте или на сервер.
- Обогащение данными из внешних сетевых сервисов.
- Анализ произвольных текстовых логов, ETL, EML или дампов памяти.

## 3. Пользовательский сценарий

1. Пользователь запускает `EVTXAuditor.exe`.
2. Приложение подставляет исходную папку `C:\Users\impal\Downloads\АУДИТ 2`, если она существует.
3. Пользователь выбирает исходную папку и каталог сохранения отчёта.
4. В окне явно показаны параметры: 30 дней, Critical/Error, опасные события ИБ и единый HTML.
5. Пользователь нажимает «Начать проверку».
6. Рабочий поток последовательно обрабатывает узлы, а окно показывает текущий архив, журнал, число обработанных узлов и диагностические сообщения.
7. Пользователь может отменить анализ. Отмена происходит между записями и этапами без повреждения исходных данных.
8. После завершения приложение сохраняет HTML, показывает краткий итог и открывает отчёт системным браузером.
9. Если отдельный источник повреждён, приложение завершает остальные узлы и отражает неполноту данных в отчёте.

## 4. Архитектура

Приложение разделяется на независимые компоненты:

1. **PySide6 GUI** — выбор каталогов, запуск, отмена, прогресс и итоговый статус.
2. **Обнаружение источников** — рекурсивный поиск ZIP, определение предполагаемого имени узла и перечисление EVTX.
3. **Безопасное извлечение** — извлечение только EVTX во временный каталог с защитой от выхода пути за его границы.
4. **EVTX-парсер** — потоковое чтение событий через `python-evtx`, разбор XML и нормализация полей.
5. **Движок правил** — классификация Critical, Error и событий ИБ.
6. **Агрегатор** — ограничение периодом, корреляция массовых попыток входа и объединение повторов.
7. **Генератор отчёта** — создание автономного HTML со встроенными стилями, JavaScript и данными.
8. **Сборка** — PyInstaller создаёт оконный x64 EXE без требования установленного Python.

GUI зависит только от публичного интерфейса координатора анализа. Парсер не зависит от GUI или HTML. Правила принимают нормализованные события. Генератор отчёта принимает готовую модель результата и не читает архивы напрямую.

## 5. Модель данных

### Нормализованное событие

Каждое прочитанное событие содержит:

- имя узла;
- имя архива и EVTX-файла;
- канал;
- поставщика;
- Event ID и при наличии Qualifiers;
- штатный уровень Windows;
- время в UTC и локальное представление;
- Record ID;
- поле Computer;
- Task, Opcode и Keywords;
- именованные поля EventData и UserData;
- доступный текст RenderingInfo;
- диагностические признаки неполного разбора.

### Найденная проблема

Проблема содержит:

- категорию `critical`, `error` или `security`;
- внутренний уровень приоритета;
- русское название;
- объяснение риска;
- рекомендации по проверке;
- ключ группировки;
- количество повторов;
- первую и последнюю временную отметку;
- список минимальных доказательств каждого повторения;
- исходные технические поля.

### Результат узла

Для каждого узла сохраняются:

- статус `checked`, `no_findings`, `partial` или `failed`;
- фактический анализируемый интервал;
- перечень обработанных архивов и журналов;
- счётчики исходных и отобранных событий;
- сгруппированные проблемы;
- предупреждения и ошибки обработки.

## 6. Определение периода

Для каждого узла период рассчитывается независимо:

1. Во время потокового чтения трёх журналов определяется максимальное корректное время события.
2. Нижняя граница равна максимальному времени минус 30 календарных суток.
3. Кандидаты на проблему сохраняются в компактной форме в процессе чтения.
4. После определения максимального времени отбрасываются кандидаты за пределами периода.
5. Если корректных временных меток нет, узел получает статус `partial` или `failed`, а приложение не подменяет отсутствие данных выводом «проблем не найдено».

Такой алгоритм не зависит от текущего времени компьютера, времени изменения ZIP или даты в имени архива.

## 7. Правила классификации

### Критические события

К категории Critical относятся:

- события со штатным уровнем Windows `1`;
- Security Event ID 1102 — очистка журнала аудита;
- System Event ID 104 от `Microsoft-Windows-Eventlog` — очистка журнала;
- изменения, отключающие аудит безопасности;
- отключение активной защиты Microsoft Defender;
- обнаружение вредоносного ПО без подтверждённого успешного устранения в доступной последовательности событий.

### Ошибки

К категории Error относятся:

- события со штатным уровнем Windows `2`;
- BugCheck, ошибки дисков, файловой системы, драйверов, приложений и служб, если их Event ID присутствует в каталоге правил;
- повторяющиеся аварийные завершения служб и приложений.

### События информационной безопасности

Каталог первой версии включает:

- 4625 — неудачный вход;
- 4719 — изменение политики аудита;
- 4720 и 4726 — создание и удаление учётной записи;
- 4728, 4732 и 4756 — добавление в группы безопасности;
- 4738 — изменение учётной записи;
- 4740 — блокировка учётной записи;
- 4771 — ошибка предварительной проверки Kerberos;
- 7045 — установка новой службы;
- события Microsoft Defender об обнаружении, устранении, отключении защиты и изменении конфигурации.

Каталог реализуется как данные и отдельные обработчики корреляции, чтобы новые правила добавлялись без изменения EVTX-парсера.

### Массовые ошибки входа

Event ID 4625 становится тревогой, если за скользящее 15-минутное окно обнаружено не менее 10 событий с общей учётной записью или общим адресом источника. В отчёте показываются:

- количество попыток;
- начало и конец окна;
- учётная запись;
- адрес источника;
- тип входа;
- Status и SubStatus;
- список Record ID и временных отметок.

Одиночные события 4625 сохраняются в статистике обработки, но не выводятся как отдельная критическая проблема.

## 8. Группировка повторов

Обычные Critical и Error объединяются по:

- узлу;
- каналу;
- поставщику;
- Event ID;
- нормализованному смыслу сообщения;
- ключевым полям, определённым правилом.

Из изменяемых частей сообщения удаляются временные метки, Record ID и несущественные числовые значения. Группировка не должна объединять разные учётные записи, адреса источника, диски, службы или исполняемые файлы, если эти поля существенны для расследования.

В HTML группа отображается одной карточкой, но встроенные данные сохраняют минимальные доказательства всех повторений. Большие списки отрисовываются JavaScript только после раскрытия, чтобы начальная загрузка отчёта оставалась быстрой.

## 9. Человекочитаемые описания

`python-evtx` гарантированно предоставляет XML события, но архивный EVTX не всегда содержит готовый текст поставщика. Поэтому используется следующий порядок:

1. RenderingInfo из события, если оно присутствует.
2. Русское название, объяснение и рекомендация из встроенного каталога правил.
3. Для неизвестного Error — поставщик, Event ID и структурированные EventData/UserData.

Приложение не должно показывать выдуманное описание. При отсутствии шаблона в отчёте явно указывается: «Описание поставщика недоступно; показаны технические поля события».

## 10. HTML-отчёт

Отчёт является одним автономным UTF-8 HTML-файлом и не требует Интернета.

### Общий уровень

- дата создания;
- исходная папка;
- число узлов, архивов и журналов;
- фактический общий охват дат;
- карточки Critical, Error, событий ИБ и проверенных узлов;
- таблица приоритета узлов;
- поиск по узлу, Event ID, поставщику и тексту;
- фильтры по серьёзности и журналу.

### Уровень узла

- имя и статус узла;
- фактический 30-дневный период;
- обработанные Application, Security и System;
- счётчики категорий;
- карточки сгруппированных проблем;
- число повторов, первая и последняя отметка;
- объяснение возможных причин;
- рекомендации по ручной проверке;
- раскрываемые исходные события и технические поля.

### Контроль качества

- статус каждого ZIP и EVTX;
- количество прочитанных записей;
- ошибки XML и повреждённые сегменты;
- отсутствующие журналы;
- явное различие между «проблемы не найдены» и «данных недостаточно».

Навигация слева содержит общую картину, критические события, события ИБ, список узлов и раздел контроля качества.

## 11. Обработка ошибок и безопасность

- ZIP открываются только для чтения.
- Извлекаются только обычные файлы с расширением `.evtx`.
- Нормализованный путь извлечения обязан оставаться внутри созданного временного каталога.
- Один EVTX ограничивается 512 МиБ, суммарный извлечённый объём одного архива — 2 ГиБ, количество записей ZIP — 1000.
- Ошибка одного события, журнала или узла записывается в диагностику и не останавливает остальные узлы.
- Временный каталог удаляется в блоке гарантированной очистки после завершения или отмены.
- Частичный HTML сначала пишется во временный файл и атомарно переименовывается только после успешного завершения генерации.
- В HTML экранируются все строки из журналов. Данные не вставляются как исполняемый код.
- В приложении отсутствуют сетевые запросы.

## 12. Интерфейс PySide6

Главное окно содержит:

- заголовок «EVTX Аудитор событий Windows»;
- поле исходной папки и кнопку «Выбрать»;
- поле папки отчёта и кнопку «Выбрать»;
- карточки активных параметров анализа;
- кнопку «Начать проверку»;
- кнопку «Отменить», активную только во время работы;
- общий индикатор выполнения;
- текущий узел и журнал;
- прокручиваемый журнал диагностических сообщений;
- итоговую панель с путём к HTML и кнопками «Открыть отчёт» и «Открыть папку».

Анализ выполняется в рабочем `QThread`; GUI не блокируется. Сигналы передают прогресс, сообщения, успешный результат, отмену и ошибку верхнего уровня.

## 13. Выходные материалы

Пользователю передаются:

- `EVTXAuditor.exe` — оконная x64-сборка;
- `EVTXAuditor-Source.zip` — полный исходный код, тесты, документация и файл зависимостей;
- каталог исходного проекта;
- сгенерированный отчёт по фактическим архивам из `АУДИТ 2`.

Имя отчёта: `EVTX_Audit_YYYY-MM-DD_HH-MM-SS.html`.

## 14. Проверка

Автоматические тесты должны покрывать:

- обнаружение ZIP и EVTX;
- защиту от небезопасных путей ZIP;
- разбор XML с EventData, UserData и неполными полями;
- расчёт 30-дневного периода;
- штатные Critical и Error;
- все правила ИБ из каталога;
- порог 4625: ниже 10, ровно 10 и несколько независимых окон;
- группировку и разделение событий по существенным полям;
- HTML-экранирование;
- различие статусов `no_findings`, `partial` и `failed`;
- генерацию автономного HTML;
- отмену анализа и очистку временных файлов.

Интеграционная проверка выполняется на копиях всех 22 реальных архивов. Проверяется:

- обнаружено 22 узла и 66 EVTX;
- исходные архивы не изменились по SHA-256;
- отчёт открывается без сетевых ресурсов;
- в отчёте присутствует каждый узел;
- ошибки отдельных источников отражены в контроле качества;
- EXE запускается без установленного Python и создаёт отчёт.

## 15. Критерии готовности

Работа считается завершённой, когда:

1. GUI запускает и отменяет анализ без зависания.
2. Все доступные архивы обрабатываются, а проблемы отдельных файлов не прерывают общий запуск.
3. Для каждого узла используется подтверждённый 30-дневный интервал.
4. Critical, Error и утверждённые ИБ-правила отражаются в едином HTML.
5. Повторы сгруппированы, но доказательства не потеряны.
6. HTML соответствует утверждённому подробному макету и работает автономно.
7. Тесты проходят.
8. Реальные 22 архива проверены, а их хеши до и после совпадают.
9. Готовый EXE проходит запуск и создание отчёта.
10. Полный исходный код и инструкции по сборке переданы вместе с EXE.
